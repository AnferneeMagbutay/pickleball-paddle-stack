## Context

The app is a single-organizer, offline, local-state PWA. The engine (`src/lib/engine.ts`) is a set of pure functions over a single `AppState`, persisted to `localStorage` and well covered by tests. These three additions must preserve that shape: pure, testable, and the existing invariant that statistics never influence queue ordering.

## Goals / Non-Goals

**Goals:**
- One-step, deterministic undo of the most recent game end (winner or void).
- Bench/resume a player without losing them or their stats.
- Restyle to Midnight Violet with no behavior change.

**Non-Goals:**
- Multi-step undo / full history. Numeric scores. Smarter shuffle. Export/backup. Saved roster. Session reset. (All deferred.)

## Decisions

### Undo via a single `lastGameEnd` snapshot, invalidated by any other mutation
- `AppState` gains `lastGameEnd: GameEndRecord | null`, where `GameEndRecord = { courtId, teamA, teamB, outcome }`.
- `endGame` sets `lastGameEnd`. `undoLastGameEnd` consumes it (then sets it to `null`).
- **Every other mutating function sets `lastGameEnd = null`** (add/remove/move/bench/resume/startGame/setCourtCount).
- **Why:** this guarantees that whenever undo is available, the state is exactly as `endGame` left it — the four players are still at the back of the queue and the court is still open. Reversal is therefore unambiguous: restore the court to `playing` with the saved teams, drop those four ids from the queue, and roll back stats. It also gives the UI a precise enable condition (`lastGameEnd !== null`) and matches the real use case (fix a mistap immediately).
- **Alternatives considered:** a full undo stack (rejected — scope; multi-step deferred). Position-based "remove last 4 from queue" without invalidation (rejected — a player added between end and undo would break it). Identity-based removal is used regardless, for clarity.

### Undo reversal details
- Court: `status → 'playing'`, `teamA`/`teamB` restored from the record.
- Queue: remove the four ids (`teamA ∪ teamB`) by identity.
- Stats: if `outcome` was `'A'`/`'B'`, decrement winners by `{wins:1, games:1}` and losers by `{losses:1, games:1}`; if `outcome` was `null` (void), no stat change. This is the exact inverse of `endGame`, keeping stats decoupled from the queue.

### Bench via a separate `benched` list
- `AppState` gains `benched: PlayerId[]`. Every roster player is in exactly one place: `queue`, a court, or `benched`.
- `benchPlayer(id)`: only a **queued** player may be benched (on-court → blocked, mirroring removal; already-benched → no-op). Moves the id from `queue` to `benched`. Stats retained.
- `resumePlayer(id)`: moves the id from `benched` to the **back** of `queue`.
- `removePlayer` is extended to also drop the id from `benched` (a benched player can be removed; on-court still blocked). Stats retained.
- `startGame` already pulls only from `queue`, so benched players are naturally excluded from rotation with no extra logic.

### Small-group warning counts only available players
- Available = `players.length − benched.length` (i.e. queue + on court). Warning when `available ≥ 1 && available < 4*courts + 4`. Benched players are not in rotation, so they don't count toward avoiding back-to-back games.

### State version bump and persistence
- Bump `CURRENT_VERSION` 1 → 2 (shape adds `benched` and `lastGameEnd`). On load, a mismatched version returns a clean initial state — the existing behavior. A one-evening session makes this acceptable; no migration needed.

### Midnight Violet restyle
- `styles.css`: swap the CSS custom properties to the Midnight Violet palette; header, primary buttons, and position chips use a `135deg` indigo→violet gradient (`#6366f1 → #8b5cf6`).
- `index.html`: `theme-color` and status-bar meta to the dark palette.
- `vite.config.ts`: manifest `theme_color`/`background_color` → `#0b1020` (and the violet accent where appropriate).
- `scripts/generate-icons.mjs`: change the icon background from teal to the violet palette and regenerate.
- No DOM/structure changes — purely tokens and assets, so existing behavior and tests are unaffected.

## Risks / Trade-offs

- **Undo is single-step and lost after any other action** → Mitigation: this is intentional and surfaced in the UI (the control only shows when reversible). Multi-step is explicitly deferred.
- **Version bump discards old saved state** → Mitigation: acceptable per existing policy; sessions are short-lived. Documented.
- **Benched player referenced by a stale `lastGameEnd`** → Not possible: benching clears `lastGameEnd`.
- **Icon regeneration drift** → Mitigation: icons are generated by the committed script; regenerate and commit the PNGs together.
